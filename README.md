# concurrency-lab

## № 1 Эксперименты с параллелизмом в Python: сравнение однопоточного, многопоточного и многопроцессорного подходов.

В данном эксперименте сравниваются три подхода к выполнению задач:
- **Однопоточность** - последовательное выполнение
- **Многопоточность** - использование нескольких потоков в одном процессе
- **Многопроцессорность** - использование нескольких процессов

### CPU-bound задания
|       Подход        |  Время  |
|---------------------|---------|
| Однопоточность      |  2.22 с |
| Многопоточность     |  2.18 с |
| Многопроцессорность |  1.13 c |

**Вывод:** По результатам было получено, при CPU-bound задачах многопроцессорность выполняет вычислительные задания быстрее, чем однопоточность примерно в 2 раз, многопроцессорность быстрее для CPU-bound задач, потому что каждый процесс имеет свой собственный GIL, и процессы выполняются параллельно на разных ядрах процессора. В многопоточности GIL не позволяет потокам выполняться одновременно на CPU, поэтому время почти не отличается от однопоточного подхода.

### I/O-bound задания
|       Подход        |  Время  |
|---------------------|---------|
| Однопоточность      |  9.59 с |
| Многопоточность     |  2.84 с |
| Многопроцессорность |  3.12 с |

**Вывод:**  Для I/O-операций многопоточность и многопроцессорность дают схожий результат, но потоки потребляют меньше памяти

## Общие выводы по эксперименту №1

- **CPU-bound**:  Эксперимент показал, что однопоточность и многопоточности схожи во времени выполнении разница в 0.02 секунды, а многопроцессорность показала себя идеально и быстрее **в 2 раза**.
- **I/O-bound**:  Многопоточность и многопроцессорность выдали близкие значения, но в I/O-bound заданиях проще будет использовать так как потоки потребляют меньше памяти, чем многопроцессорность.
- В многопоточности GIL не проблема для I/O (блокировка освобождается при ожидании), но тормозит CPU-вычисления. В многопроцессорности GIL не проблема для CPU-вычислений, у каждого процесса свой GIL и каждый работает не мешая другому.


## № 2 Эксперименты с параллелизмом в Python: асинхронное программирование, синхронное программирование при запросе к локальному серверу 50 раз.

В данном эксперименте сравниваются три подхода к выполнению задач:
- **Асинхронный без семафора** - асинхронное выполнение кода без ограничения одновременных запросов
- **Асинхронный с семафором** - асинхронное выполнение кода с ограничения одновременных запросов
- **Синхронный** - последовательное выполнение запросов

### I/O-bound задания с неблокирующей задержкой
|         Подход           | Запросы |   Время    |
|--------------------------|---------|------------|
| Асинхронный без семафора |  50/50  |    1.31 с  |
| Асинхронный с семафором  |  50/50  |   11.68 с  |
| Синхронный               |  50/50  |   52.61 с  |

### I/O-bound задания с блокирующей задержкой
|         Подход           | Запросы |   Время    |
|--------------------------|---------|------------|
| Асинхронный без семафора |  50/50  |   50.11 с  |
| Асинхронный с семафором  |  50/50  |   50.11 с  |
| Синхронный               |  50/50  |   52.22 с  |

### Влияние семафора
|        Параметр        | Без семафора | С семафором |
|------------------------|--------------|-------------|
| Время выполнения       |     1.31с    |   11.68с    |
| Одновременных запросов |      50      |     10      |
| Нагрузка на сервер     |    Высокая   |    Низкая   |
|      Скорость          |    Быстро    |   Медленно  |

**Вывод:** Семафор - это компромисс между скоростью и нагрузкой на сервер. Его необходимо нужно использовать, когда нужно защитить внешний API от перегрузки.

## Общие выводы по эксперименту №2 
- При использование семафора мы ограничиваем, нагрузку на сервер, но при этом увеличиваем время выполнения, эксперимент показал что асинхронный подход с семафором выполняется дольше чем без семафора **в 8.9 раз** почти, такая большая разница объясняется тем что в эксперименте без семафора запросы ушли все одновременно, а у эксперимента с семафором там было ограниченное количество одновременных запросов, но с меньшей нагрузкой на сервер.
- Эксперимент показал что асинхронный подход с aiohttp сессией быстрее почти **в 40 раз**, чем синхронный с requests это можно объяснить тем, что задачи переключаются в потоке.
- Эксперимент **I/O-bound задания с блокирующей задержкой показал**, что все подходы показывают одно и тоже время, так как в корутине используется 'time.sleep()' синхронный метод который заблокирует поток вместе со всем event loop: цикл встанет, ни одна другая задача крутиться не будет. 


## № 3 Эксперимент потоки vs процессы vs asyncio в Python: использование многопоточноти с использованием модуля 'ThreadPoolExecutor', синхронное программирование, асинхронное программирование при запросе к локальному серверу 100 раз.

В данном эксперименте сравниваются четыре подхода к выполнению задач:
- **ThreadPoolExecutor** - использование пула потоков для параллельного выполнения
- **Асинхронный с семафором** - неблокирующее выполнение с event loop с ограничением одновременных запросов
- **Асинхронный без семафором** - неблокирующее выполнение с event loop без ограничения на одновременные запросы
- **Синхронный** - последовательное выполнение запросов

### I/O-bound задания
|         Подход           |  Запросы  |   Время        |
|--------------------------|-----------|----------------|
| ThreadPoolExecutor       |  100/100  |     61.38  с   |
| Асинхронный с семафором  |  100/100  |     22.67  с   |
| Асинхронный без семафором|  100/100  |      1.31  с   |
| Синхронный               |  100/100  |     103.22 с   |


### Потоки vs процессы vs asyncio
|        Параметры          |  threading   | multiprocessing|    asyncio   |
|---------------------------|--------------|----------------|--------------|
| Параллелизм/конкурентность|конкурентность|  параллелизм   |конкурентность|
|   CPU-bound/I/O-bound     |  I/O-bound   |   CPU-bound    |   I/O-bound  |
|    Накладные расходы      |    Высокие   |    Высокие     |    Низкие    |
|    Потребление памяти     |     Малое    |     Большое    |     Малое    |
|        Обход GIL          |     Нет      |      Да        |      Нет     |

## Общие выводы по эксперименту №3 
- **Асинхронность без семафора** — показал самый высокий результат, быстрее синхронного подхода в **78 раз**. Это достигается благодаря за счет переключения задач в event loop.
- **Асинхронность с семафором** медленнее без семафора в **17 раз**, но позволяет контролировать нагрузку на сервер ограничевая одновременные запросы.
- **ThreadPoolExecutor** уступает асинхронности с семафором в **2.7 раза** из-за накладных расходов на создание и переключение потоков. 
- **Синхронный подход** ожидаемо самый медленный, так как запросы выполняются последовательно.
-  Данные эксперименты показали, что для I/O-bound задач лучше использовать асинхронность либо многопоточность, но из-за высоких накладных расходов (создание потоков, переключение между потоками, на GIL) лучше использовать асинхронность там все происходит в рамках одного потока и event loop крутит задачи в этом потоке, в тот момент когда задача ожидает ответ.
